# 피그말리온 루프 (Pygmalion Loop) — 자기진화 워크플로우

> 피그말리온은 에이전트가 아닙니다. 실시간 자기변형은 금지됩니다.
> 관찰 → 패턴 탐지 → 개선안 생성 → 그림자 평가 → 승격/폐기의 **통제된 진화**만 수행합니다.

---

## 1. 개요

### 목적
AI 에이전트와 스킬의 품질을 **시간이 지남에 따라 자동으로 개선**합니다. 반복되는 실패 패턴, 비효율적 프로세스, 새로운 도메인 요구를 감지하여 시스템을 점진적으로 진화시킵니다.

### 핵심 원칙
- **실시간 자기변형 금지**: 작업 중에 스킬/에이전트를 실시간으로 수정하지 않음
- **통제된 진화**: 관찰 → 후보 생성 → 시험 → 승격 4단계를 반드시 거침
- **근거 기반**: 개선은 반드시 관찰 데이터(실패 기록, 사용 패턴)에 기반
- **비파괴적**: 기존 스킬/에이전트를 삭제하지 않고, 새 버전을 시험 후 교체

---

## 2. 트리거 조건

피그말리온은 아래 조건을 감지했을 때만 작동합니다. 조건 미충족 시 비활성.

| 트리거 | 조건 | 의미 |
|--------|------|------|
| **반복 수정** | 같은 유형의 수정 3회 이상 / 30일 이내 | 스킬이 해당 패턴을 사전에 방지하지 못함 |
| **반복 흐름** | 동일 작업 흐름 5회 이상 / 60일 이내 | 자동화 또는 템플릿화 기회 |
| **새 도메인** | 기존에 없는 도메인에서 3개 프로젝트 수행 | 새 스킬/에이전트 필요 가능성 |
| **랄프 루프 실패** | 기획 재검토 판정 2회 이상 / 90일 이내 | 근본적 품질 문제 |
| **사용자 명시 요청** | "이 부분 개선해줘" 등 | 수동 트리거 (항상 유효) |

## 3. 5단계 프로세스

### 3-1단계: 관찰 (Observe)

4가지 데이터를 수집합니다:

| 데이터 유형 | 수집 내용 | 저장 위치 |
|------------|----------|----------|
| **수정 기록** | 랄프 루프에서 Momus가 지적한 항목들 | 프로젝트 docs/ |
| **사용 패턴** | 어떤 에이전트/스킬이 얼마나 자주 호출되는지 | 워크스페이스 로그 |
| **실패 기록** | 기획 재검토 판정 이유, 반복 실패 유형 | 프로젝트 docs/ |
| **신규 요구** | 기존 스킬로 커버되지 않는 요청 유형 | 사용자 요청 로그 |

#### 관찰 기록 형식
```markdown
## Pygmalion 관찰 기록

### 기간: YYYY-MM-DD ~ YYYY-MM-DD
### 수집 데이터

#### 반복 수정 패턴
| 수정 유형 | 발생 횟수 | 관련 스킬 | 예시 |
|----------|----------|----------|------|
| 표 출처 누락 | 4회 | consulting-report | P2026-002 최종보고서 |
| NCS 코드 오류 | 3회 | job-analysis | P2026-006 직무분석 |

#### 반복 흐름
| 흐름 | 발생 횟수 | 관련 에이전트 |
|------|----------|-------------|
| 설문설계→분석→보고서 | 6회 | Hera→Apollo→Hermes |

#### 실패 기록
| 산출물 | 실패 원인 | 관련 스킬/에이전트 |
|--------|----------|------------------|
| [산출물명] | [원인] | [스킬/에이전트] |
```

### 3-2단계: 패턴 탐지 (Detect)

관찰 데이터에서 개선 가능한 패턴을 식별합니다:

| 패턴 유형 | 식별 방법 | 개선 방향 |
|----------|----------|----------|
| **반복 지적** | 같은 Momus 지적사항 3회+ | 스킬에 해당 규칙 추가 |
| **누락 패턴** | 특정 항목 반복 누락 | 체크리스트 보강 |
| **비효율 흐름** | 같은 에이전트 조합 반복 | 워크플로우 템플릿화 |
| **미지원 도메인** | 기존 스킬 매칭 실패 | 신규 스킬 후보 생성 |
| **품질 하향** | 등급 A 비율 감소 추세 | 스킬 점검 필요 |

### 3-3단계: 개선안 생성 (Generate)

패턴별 개선안을 구체적으로 작성합니다:

#### 개선안 유형
| 유형 | 설명 | 예시 |
|------|------|------|
| **스킬 보강** | 기존 스킬에 규칙/체크리스트 추가 | consulting-report에 "표 출처 필수" 규칙 추가 |
| **스킬 신규** | 새 도메인용 스킬 생성 | ESG-consulting 스킬 신규 작성 |
| **에이전트 개선** | 에이전트 프롬프트 보강 | Hera에 최신 노동법 참조 추가 |
| **워크플로우 추가** | 반복 흐름 자동화 | 설문→분석→보고 워크플로우 |
| **품질계약 개정** | 품질 기준 추가/변경 | 신규 산출물 유형 기준 추가 |

#### 개선안 형식
```markdown
## Pygmalion 개선안

### 개선안 ID: PYG-YYYY-NNN
### 트리거: [트리거 조건]
### 관찰 근거: [관찰 데이터 참조]

### 대상
- 파일: `_shared/skills/consulting-report/SKILL.md`
- 변경 유형: 스킬 보강

### 현재 상태
[현재 스킬에서 부족한 부분]

### 제안 변경
[구체적 변경 내용 — 추가할 규칙, 수정할 프로세스 등]

### 기대 효과
[이 변경으로 해결되는 반복 문제]

### 리스크
[변경으로 발생할 수 있는 부작용]
```

### 3-4단계: 그림자 평가 (Shadow Evaluate)

개선안을 **실제 적용하기 전에** 시험합니다:

1. **시뮬레이션**: 과거 프로젝트에 개선안을 적용했을 때 결과가 개선되는지 확인
   - 과거 실패 사례에 개선된 스킬을 적용 → 문제가 해결되는가?
2. **충돌 검사**: 다른 스킬/에이전트와 충돌하지 않는지 확인
   - 새 규칙이 기존 규칙과 모순되지 않는가?
3. **범위 확인**: 개선이 의도하지 않은 영향을 미치지 않는지 확인

#### 평가 기준
| 기준 | 통과 조건 |
|------|----------|
| 효과성 | 과거 실패 사례의 80% 이상 해결 예상 |
| 무해성 | 기존 기능 저하 없음 |
| 일관성 | 다른 스킬/에이전트와 모순 없음 |
| 실용성 | 구현 가능하고 유지보수 부담 적음 |

### 3-5단계: 승격 또는 폐기 (Promote or Discard)

#### 승격 (3개 기준 모두 충족 시)
1. 개선안을 `_shared/` 원본에 적용
2. `sync-tools.cmd`로 전 프로젝트에 배포
3. 변경 이력을 REGISTRY.md에 기록
4. Git commit & push

#### 폐기 (1개 기준이라도 미충족 시)
1. 개선안을 기록만 남기고 적용하지 않음
2. 폐기 사유 기록 (향후 재검토 가능)

## 4. 거버넌스

### 자동 vs 수동
| 변경 규모 | 승인 방식 |
|----------|----------|
| **소규모** (체크리스트 항목 1-2개 추가) | 자동 승격 가능 |
| **중규모** (스킬 섹션 추가/수정) | 사용자 확인 후 승격 |
| **대규모** (에이전트 재설계, 신규 스킬) | 사용자 명시적 승인 필수 |

### 변경 빈도 제한
- **월 최대 3건**: 과도한 변경으로 시스템 불안정 방지
- **변경 간 최소 간격**: 7일 (이전 변경의 효과 관찰 기간)
- **긴급 수정**: 시스템 오류 수정은 빈도 제한 예외

### 롤백
- 모든 변경은 Git에 기록됨 → 문제 발생 시 `git revert`로 롤백
- 롤백 판단 기준: 변경 후 품질 등급이 이전보다 하락하면 롤백

## 5. 연계 체계

```
[랄프 루프] ──→ 실패 데이터 ──→ [Pygmalion 관찰]
                                      │
                                      ▼
                              [패턴 탐지 → 개선안]
                                      │
                                      ▼
                              [그림자 평가 → 승격]
                                      │
                                      ▼
                              [_shared/ 업데이트]
                                      │
                                      ▼
                              [sync-tools.cmd 배포]
                                      │
                                      ▼
                              [다음 산출물에서 효과 관찰]
```

| 참조 문서 | 역할 |
|----------|------|
| `_shared/workflows/ralph-loop.md` | 실패 데이터 공급원 |
| `_shared/quality-contracts/QUALITY_CONTRACT.md` | 품질 기준 원본 (개선 대상) |
| `_shared/REGISTRY.md` | 변경 이력 기록 |

## 6. 사용 방법

피그말리온은 **트리거 조건 충족 시 자동 제안**됩니다.

- 평소: 비활성 (관찰만 수행)
- 트리거 감지: "Pygmalion 개선안이 있습니다" 알림
- 사용자가 "개선해줘" 직접 요청: 수동 트리거

**사용자가 할 일**: 중규모 이상 변경 시 승인/거부 판단만 하면 됩니다.
