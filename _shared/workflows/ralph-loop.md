# 랄프 루프 (Ralph Loop) — 품질 검증 워크플로우

> "팀장이 개입해야 한다면, 시스템이 실패한 것"
> 랄프 루프는 AI 산출물이 시니어 컨설턴트 수준을 보장하는 자동 품질 게이트입니다.

---

## 1. 개요

### 적용 범위
- **모든 납품용 산출물**에 자동 적용 (보고서, 제안서, 분석 결과, 직무기술서 등)
- 내부 작업 메모, 탐색 결과, 중간 데이터는 대상 아님
- 리드 에이전트가 산출물 생성을 완료하면 자동으로 랄프 루프 진입

### 핵심 원칙
- **최대 2회 수정**: 2회 이내에 품질계약 통과 못하면 기획 단계 문제
- **검증자 분리**: 생산자(리드 에이전트)와 검증자(Momus)는 반드시 분리
- **근거 기반 피드백**: "좋지 않다"가 아닌, "품질계약 제X조 위반: 구체적 내용" 형식
- **즉시 반영**: 피드백 수신 즉시 수정, 지연 금지

## 2. 프로세스 흐름

```
[리드 에이전트] 초안 생성
         │
         ▼
   ┌─────────────┐
   │ Momus 1차 검증 │ ← 품질계약(QUALITY_CONTRACT.md) 기준 적용
   └─────┬───────┘
         │
    통과? ─── Yes ──→ ✅ 확정 (등급 A)
         │
        No
         │
         ▼
   [리드 에이전트] 1차 수정
         │
         ▼
   ┌─────────────┐
   │ Momus 2차 검증 │ ← 1차 지적사항 해소 여부 + 전체 재확인
   └─────┬───────┘
         │
    통과? ─── Yes ──→ ✅ 확정 (등급 B)
         │
        No
         │
         ▼
   ⚠️ 기획 재검토 (Brief 보완 필요)
   → 사용자에게 보고: "산출물이 2회 수정에도 품질 기준 미달.
     원인: [구체적 원인]. 기획 단계에서 [보완 필요 사항] 확인 후 재시작 권장."
```

## 3. 단계별 상세

### 3-1. 초안 생성 (리드 에이전트)
- 담당 에이전트가 관련 스킬을 활용하여 산출물 초안 작성
- 초안 완성 시 자동으로 검증 단계 진입
- **초안 품질 기준**: 품질계약 공통 5항목을 스스로 1차 점검한 후 제출
  - 정확성, 완결성, 일관성, 가독성, 근거성

### 3-2. Momus 1차 검증
**검증 기준**: `_shared/quality-contracts/QUALITY_CONTRACT.md`

#### 검증 체크리스트
```
[공통 기준 — 5항목]
□ 정확성: 데이터, 수치, 인용이 사실에 부합하는가
□ 완결성: 요청된 모든 항목이 빠짐없이 포함되었는가
□ 일관성: 용어, 형식, 톤이 문서 전체에서 통일되었는가
□ 가독성: 비전문가도 즉시 이해할 수 있는가
□ 근거성: 모든 주장에 데이터·이론·사례 근거가 있는가

[유형별 기준 — 해당 유형 전항목]
□ (pptx) 스토리라인, 1슬라이드1메시지, 시각비율, 폰트/색상 통일
□ (docx) 프레임워크, 표/그림 번호, 목차 일치, 참고문헌
□ (xlsx) 계산 검증, 시트 구성, 서식 통일, 단위 명시
□ (제안서) RFP 반영도, 방법론 도식, 간트차트, 유사 실적
□ (직무분석) NCS 매핑, FID 완비, 역량 수준, 직무기술서 항목
```

#### 검증 결과 형식
```
## Momus 검증 보고 (1차)

**산출물**: [산출물명]
**생산 에이전트**: [에이전트명]
**검증 일시**: YYYY-MM-DD HH:MM
**품질 등급**: A / B / C / D

### 합격 항목
- ✅ 정확성: 모든 수치 출처 확인
- ✅ 일관성: 용어 통일

### 불합격 항목 (수정 필요)
- ❌ 완결성: [구체적 누락 내용]
  → 수정 지시: [정확한 수정 방법]
- ❌ 근거성: [근거 없는 주장 위치]
  → 수정 지시: [필요한 근거 유형]

### 종합 판정
- [ ] 통과 → 확정
- [x] 수정 필요 → 1차 수정 지시
- [ ] 재작성 필요 → 기획 재검토
```

### 3-3. 1차 수정 (리드 에이전트)
- Momus 피드백의 **모든 지적사항**을 빠짐없이 반영
- 수정 완료 후 변경 내역을 명시: "지적사항 3건 중 3건 수정 완료"
- **범위 제한**: 지적사항만 수정. 다른 부분 임의 변경 금지

### 3-4. Momus 2차 검증
- **1차 지적사항 해소 여부** 최우선 확인
- 해소 확인 후 전체 문서 재점검 (1차 수정으로 인한 새 문제 발생 여부)
- **결과**:
  - 통과 → 등급 B로 확정
  - 미통과 → 기획 재검토 판정

### 3-5. 기획 재검토 (실패 시)
2회 수정에도 품질계약 미달 시:
1. **원인 분석**: 왜 2회 만에 통과하지 못했는가
   - 요구사항 불명확 (brief 부족)
   - 데이터 부족 (수집 미완료)
   - 스킬 부족 (해당 스킬 개선 필요 → Pygmalion 트리거)
2. **사용자 보고**: 원인 + 보완 필요 사항 + 재시작 조건
3. **Pygmalion 기록**: 실패 패턴을 Pygmalion 관찰 데이터에 추가

## 5. 적용 예외

### 랄프 루프 생략 가능한 경우
- 사용자가 명시적으로 "검증 생략" 요청한 경우
- 탐색/조사 결과 (Apollo 리서치 결과 등) — 납품물이 아닌 내부 참고용
- 단순 데이터 변환/정리 (xlsx 셀 정리 등)

### 간소화 검증 (1차만 실시)
- 기존 산출물의 소규모 수정 (5% 미만 변경)
- 템플릿 기반 반복 산출물 (동일 형식, 데이터만 변경)

## 7. 사용 방법

랄프 루프는 자동 적용됩니다. 별도 호출 불필요.

1. 에이전트가 산출물을 생성하면 → 랄프 루프 자동 진입
2. Momus가 품질계약 기준으로 검증
3. 통과 시 확정, 미통과 시 수정 지시
4. 최대 2회 반복 후 확정 또는 기획 재검토

**사용자가 할 일**: 없음. 결과만 확인하면 됩니다.
