## 2. 업그레이드 요구사항 정리

### ❌ 제거할 기능
- 이메일 알림 (Nodemailer)
- 슬랙 알림 (@slack/web-api)
- 알림 설정 페이지
- Subscriber 관리 (구독자)
- NotifyLog 모델

### ✅ 유지/개선할 기능
- 입찰공고 검색 (용역만)
- 사전규격 검색 (**용역만으로 축소**)
- 발주계획 검색 (용역만)
- 키워드 모니터링
- 엑셀 내보내기
- 크롤링 (CRON)

### 🆕 신규 추가 기능

#### A. 관리자 시스템
1. **관리자 페이지** (`/admin`)
   - 관리자(이석주)가 사용자 아이디 생성/관리
   - 본부별 아이디 생성 (예: `hr본부`, `경영지원본부` 등)
   - 계정 활성화/비활성화
   - 비밀번호 초기화
2. **사용자 인증 강화**
   - 관리자가 만든 아이디로만 로그인 가능
   - 로그인하지 않으면 서비스 이용 불가 (미들웨어 보호)
3. **본부별 키워드 관리**
   - 관리자가 각 본부(아이디)별 모니터링 키워드 지정
   - 사용자는 자기 본부에 할당된 키워드 기반으로만 데이터 조회

#### B. 이용 현황 대시보드 (관리자 전용)
1. **접속 현황**
   - 사용자별 최근 접속 일시
   - 일별/주별 접속 횟수 통계
   - 활성/비활성 사용자 현황
2. **키워드별 현황**
   - 본부별 키워드 수
   - 키워드별 매칭 공고 건수
   - 신규 공고 알림 현황

#### C. 개찰결과(낙찰정보) 추가
1. **공공데이터개방표준서비스** 활용 (이미 승인됨!)
   - 용역 낙찰 정보 조회
   - 낙찰 기업명, 금액, 순위
2. **계약정보서비스** 활용 (이미 승인됨!)
   - 계약 체결 정보 조회
3. **UI**: 검색 결과에 "개찰결과" 탭 추가

#### D. 수주 분석 시스템
1. **기업별 수주 현황 분석**
   - 어느 기업이 어느 유형의 사업을 많이 수주하는지
   - 기관별 발주 → 낙찰 기업 매핑
   - 수주 금액 규모별 분석
2. **기관별 발주 패턴 분석**
   - 기관별 발주 빈도, 평균 예산
   - 주요 발주 기관 TOP N
3. **키워드 기반 분석**
   - 본부별 키워드와 매칭되는 입찰/낙찰 트렌드
   - 경쟁사(낙찰 기업) 분석

## 3. DB 스키마 변경 계획

### 기존 유지 (수정)
```prisma
model User {
  id          Int       @id @default(autoincrement())
  username    String    @unique
  password    String
  name        String              // 본부명 또는 사용자명
  department  String              // 본부/부서
  role        String    @default("user")  // "admin" | "user"
  active      Boolean   @default(true)
  lastLoginAt DateTime?           // 🆕 최근 접속 일시
  loginCount  Int       @default(0)       // 🆕 총 접속 횟수
  createdAt   DateTime  @default(now())
  keywords    Keyword[]
  accessLogs  AccessLog[]         // 🆕 접속 로그
}
```

### 신규 모델
```prisma
model AccessLog {
  id        Int      @id @default(autoincrement())
  userId    Int
  user      User     @relation(fields: [userId], references: [id])
  action    String   // "login" | "search" | "view" | "export"
  detail    String?  // 검색 키워드, 조회 항목 등
  ip        String?
  createdAt DateTime @default(now())
}

model BidResult {
  id            Int      @id @default(autoincrement())
  bidNtceNo     String   // 입찰공고번호
  bidNtceOrd    String?  // 입찰공고차수
  bidNtceNm     String   // 공고명
  opengDt       String?  // 개찰일시
  sucsfbidMthdNm String? // 낙찰방법
  companyName   String?  // 낙찰 기업명
  companyBizno  String?  // 사업자번호
  bidAmount     String?  // 입찰금액
  ranking       Int?     // 순위
  resultType    String   // "낙찰" | "유찰" | "재입찰"
  agency        String?  // 발주기관
  rawData       String
  createdAt     DateTime @default(now())

  @@unique([bidNtceNo, bidNtceOrd, companyBizno])
}

model Company {
  id           Int      @id @default(autoincrement())
  bizno        String   @unique  // 사업자번호
  name         String            // 기업명
  totalBids    Int      @default(0)  // 총 입찰 참여 수
  totalWins    Int      @default(0)  // 총 수주 수
  totalAmount  String?  // 총 수주 금액
  lastBidDate  String?  // 최근 입찰일
  createdAt    DateTime @default(now())
  updatedAt    DateTime @updatedAt
}
```

### 삭제 모델
```
- NotifyLog    (알림 로그 → 불필요)
- Subscriber   (구독자 → 불필요)
- Settings     (이메일/슬랙 설정 → 불필요)
```

## 4. 구현 단계 (Phase별)

### Phase 1: 기반 정비 & 인증 시스템 (필수/우선)

#### 1-1. 알림 기능 제거
- `lib/notify.ts` 삭제
- `app/api/notify/route.ts` 삭제
- `app/api/subscribers/route.ts` 삭제
- `app/api/settings/route.ts` 삭제
- `app/(dashboard)/settings/page.tsx` 삭제
- `lib/scheduler.ts`에서 알림 관련 코드 제거
- `package.json`에서 `nodemailer`, `@slack/web-api`, `node-cron` 제거
- DB 스키마에서 NotifyLog, Subscriber, Settings 모델 삭제

#### 1-2. 관리자 페이지 구현
- `/admin` 라우트 신설 (role="admin" 전용)
- 사용자 CRUD (생성/수정/삭제/비활성화)
- 본부별 아이디 생성 UI
- 비밀번호 초기화 기능
- 사용자 목록 (활성/비활성 필터)

#### 1-3. 인증 시스템 강화
- 로그인 시 AccessLog 기록
- lastLoginAt, loginCount 자동 업데이트
- 미들웨어에서 role 기반 접근 제어
- 관리자 초기 계정 시드 (seed.ts 수정)

#### 1-4. 본부별 키워드 관리
- 관리자: 각 사용자(본부)에 키워드 할당
- 사용자: 자기 키워드만 표시
- 키워드 페이지 권한 분리 (관리자=전체, 사용자=자기것만)

### Phase 2: 개찰결과 & 계약정보 추가

#### 2-1. 공공데이터개방표준서비스 연동
- **Base URL**: `https://apis.data.go.kr/1230000/as/ScsbidInfoService` 또는 공공데이터개방표준서비스
- 용역 개찰결과 목록 조회
- 낙찰 기업, 금액, 순위 수집
- `lib/narajan-api.ts`에 `searchBidResults()` 함수 추가

#### 2-2. 계약정보서비스 연동
- 계약 체결 정보 조회
- 입찰공고 → 계약 연결

#### 2-3. UI 확장
- 대시보드 탭에 "개찰결과" 추가
- 검색 결과에 "개찰결과" 탭 추가
- 상세보기에 낙찰자/순위 정보 표시

#### 2-4. DB 확장
- BidResult 모델 추가 (마이그레이션)
- Company 모델 추가
- 크롤링 시 개찰결과도 자동 수집

### Phase 3: 수주 분석 & 관리자 대시보드

#### 3-1. 관리자 이용 현황 대시보드
- `/admin/analytics` 페이지
- 사용자별 접속 통계 (일별/주별 차트)
- 활성 사용자 현황
- 검색 키워드 빈도 분석

#### 3-2. 수주 분석 페이지
- `/analytics` 페이지 (일반 사용자도 접근 가능)
- **기업별 수주 현황**: 기업명 → 수주 건수, 금액, 주요 기관
- **기관별 발주 패턴**: 기관명 → 발주 건수, 평균 예산, 주요 낙찰 기업
- **키워드별 트렌드**: 월별 공고/낙찰 건수 추이
- **경쟁사 분석**: 내 키워드 영역에서 많이 수주하는 기업 TOP N
- Recharts 기반 차트 (파이, 바, 라인)

#### 3-3. 데이터 집계 API
- `/api/analytics/companies` — 기업별 수주 통계
- `/api/analytics/agencies` — 기관별 발주 통계
- `/api/analytics/trends` — 시계열 트렌드
- `/api/analytics/usage` — 이용 현황 (관리자 전용)

### Phase 4: 고도화

#### 4-1. 용역 전용 최적화
- 사전규격: 물품/공사/외자 엔드포인트 제거 → 용역(Servc)만
- 불필요한 데이터 수집 중단으로 성능 향상

#### 4-2. 데이터 축적 강화
- 크롤링 기간 확대 (7일 → 90일)
- 히스토리 보존 (삭제 안 함)
- 엑셀 내보내기에 분석 데이터 포함

#### 4-3. UI/UX 개선
- 사이드바 메뉴 구조 개편 (관리자/사용자 구분)
- 반응형 개선
- 로딩/에러 상태 개선

## 5. 사이드바 메뉴 구조 (변경 후)

### 일반 사용자
```
📊 대시보드          ← 키워드 모니터링 + 통합검색
📈 수주 분석         ← 기업별/기관별/트렌드 분석
🏷️ 내 키워드        ← 본부에 할당된 키워드 확인
👤 내 정보          ← 비밀번호 변경 등
```

### 관리자
```
📊 대시보드          ← 키워드 모니터링 + 통합검색
📈 수주 분석         ← 기업별/기관별/트렌드 분석
🏷️ 키워드 관리      ← 전체 키워드 + 본부별 할당
👥 사용자 관리       ← 계정 생성/관리
📋 이용 현황         ← 접속 통계, 키워드별 현황
⚙️ 시스템 설정      ← 크롤링 주기 등
```

## 6. API 활용 전략

### 현재 승인된 API로 가능한 것

| 필요 데이터 | 활용 API | 가능 여부 |
|-----------|---------|----------|
| 용역 입찰공고 | 입찰공고정보서비스 (변경신청 중) | ✅ 기존 동작 중 |
| 용역 사전규격 | 사전규격저정보서비스 (변경신청 중) | ✅ 기존 동작 중 |
| 용역 발주계획 | 발주계획현황서비스 (승인) | ✅ 기존 동작 중 |
| 용역 개찰결과 | **공공데이터개방표준서비스** (승인) | ✅ 신규 연동 |
| 계약 체결 정보 | **계약정보서비스** (승인) | ✅ 신규 연동 |
| 세부 평가 점수 | 별도 API 없음 | ⚠️ 불가 |

### 공공데이터개방표준서비스 활용 (핵심!)
- 이미 승인되어 있어 추가 신청 불필요
- 입찰·낙찰·계약 정보를 **표준 형식**으로 통합 제공
- 용역 개찰결과 조회에 최적

### 추가 신청 권장
- **낙찰정보서비스** (15129397): 개찰순위, 복수예비가 등 상세 정보 제공
- 공공데이터개방표준서비스로 기본 낙찰정보는 커버 가능하지만, 더 상세한 순위/점수 정보를 원한다면 추가 신청 권장
