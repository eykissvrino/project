01 OhMyOpenCode란?

Claude Code를 "혼자 일하는 AI"에서 "AI 전문가 팀"으로 업그레이드하는 플러그인

뭐가 다른가요?

기본 Claude Code+ OhMyOpenCode
AI 모델Claude 하나만Claude + GPT + Gemini + Grok 자동 배합
작업 방식순차 처리여러 에이전트 병렬 처리
코드 검색직접 grep전문 검색 에이전트 자동 파견
외부 조사수동 검색Librarian이 문서/라이브러리 자동 조사
품질 검증없음Oracle/Momus가 자동 리뷰
작업 관리없음Todo 시스템으로 진행상황 추적

비유로 이해하기

기본 Claude Code = 혼자서 다 하는 프리랜서 OhMyOpenCode = 시니어 개발자(Sisyphus)가 이끄는 개발팀 시니어(Sisyphus): "로그인 페이지 만들어야 해" ├─ 인턴(Explore): "기존 코드 패턴 찾아왔습니다!" ├─ 리서처(Librarian): "라이브러리 문서 정리했습니다!" ├─ 시니어가 직접 구현 └─ CTO(Oracle): "아키텍처 검토 완료!"

02 기본 개념 이해하기

2.1 에이전트 (Agent) = AI 전문가

각각 다른 역할과 능력을 가진 AI 전문가들입니다.

Sisyphus ─── 나(총괄 매니저). 여러분이 대화하는 상대 ├── Explore ────── 코드베이스 고속 검색 (가볍고 빠름) ├── Librarian ──── 외부 문서/라이브러리 조사 (구글링 대체) ├── Oracle ─────── 아키텍처 자문/디버깅 (고급, 비쌈) ├── Metis ──────── 작업 분석/모호성 검출 (기획 단계) └── Momus ──────── 계획 리뷰/품질 검증 (QA 역할)

2.2 카테고리 (Category) = 작업 분류

작업을 맡기면 성격에 맞는 최적 AI 모델로 자동 라우팅됩니다.

"UI 만들어줘" → visual-engineering (디자인 특화 모델)

"알고리즘 짜줘" → ultrabrain (논리 추론 특화 모델)

"오타 고쳐줘" → quick (빠르고 저렴한 모델)

"문서 써줘" → writing (글쓰기 특화 모델)

2.3 스킬 (Skill) = 전문 지식 카드

특정 작업에 대한 상세한 노하우가 담긴 카드입니다. 에이전트에게 장착시키면 품질이 올라갑니다.

"Next.js 프로젝트 만들어줘"

→ nextjs15-init 스킬 자동 장착

→ 프로젝트 구조, 설정, 컨벤션까지 알고 작업

03 기본 사용법

3.1 이렇게 말하면 됩니다

OhMyOpenCode에서는 자연어로 말하면 알아서 최적의 방법을 찾아 실행합니다.

간단한 작업

"package.json에 axios 추가해줘"
"이 함수 이름 calculateTotal로 바꿔줘"
"README.md 한국어로 번역해줘"

조사/탐색

"이 프로젝트에서 인증 로직이 어디 있어?"
"React Query 사용법 알려줘"
"이 에러 왜 나는 거야? [에러 메시지]"

복잡한 작업

"로그인/회원가입 페이지 만들어줘"
"이 API를 REST에서 GraphQL로 바꿔줘"
"테스트 코드 전체 작성해줘"

3.2 잘 말하는 법 (프롬프트 팁)

❌ 나쁜 예시

"고쳐줘"

"개선해줘"

"뭔가 이상해"

✅ 좋은 예시

"src/auth/login.ts에서 비밀번호 유효성 검사가 안 되고 있어. 8자 이상 + 특수문자 필수로 바꿔줘"

"빌드하면 'Module not found' 에러 나는데, next.config.js 경로 설정 문제인 것 같아"

핵심 공식

[어디서] + [무엇이] + [어떻게] + [왜]

"src/components/Header.tsx에서(어디서) 저장 버튼이(무엇이) 편집 버튼 오른쪽에 추가되었으면 해(어떻게). 사용자가 작업 중 빠르게 저장할 수 있게(왜)."

04 에이전트 시스템 이해하기

4.1 Explore (탐험가)

역할: 코드베이스 내부를 빠르게 검색합니다.

사용자: "이 프로젝트에서 데이터베이스 연결하는 부분 어디 있어?"

→ Explore 에이전트 발사 (백그라운드)
→ "DB 연결 패턴 찾아라: connection, pool, prisma, database 키워드 검색"
→ 결과 수집 후 정리해서 답변

언제 자동 발동?

  • 코드베이스 구조를 파악해야 할 때
  • 특정 패턴이나 함수를 찾아야 할 때
  • 여러 파일에 걸친 변경이 필요할 때

4.2 Librarian (사서)

역할: 외부 문서, 라이브러리, 오픈소스 코드를 조사합니다.

사용자: "Tanstack Query v5에서 useSuspenseQuery 어떻게 써?"

→ Librarian 에이전트 발사 (백그라운드)
→ "공식 문서, GitHub 예시 코드, 베스트 프랙티스 조사해라"
→ 결과 수집 후 정리해서 답변

언제 자동 발동?

  • 외부 라이브러리 사용법이 필요할 때
  • 최신 API 문서가 필요할 때
  • 오픈소스 구현 예시가 필요할 때

4.3 Oracle (신탁)

역할: 아키텍처 설계, 어려운 디버깅 자문을 합니다. (비싸지만 정확)

사용자: "마이크로서비스로 전환하려는데 어떤 구조가 좋을까?"

→ 먼저 Explore로 현재 코드 구조 파악
→ Oracle에게 "현재 구조 분석 + 마이크로서비스 전환 전략 자문"
→ Oracle의 아키텍처 조언을 바탕으로 답변

4.4 Metis (분석가)

역할: 작업 시작 전에 모호한 점, 숨은 요구사항을 찾아냅니다.

사용자: "결제 시스템 만들어줘"

Metis 분석: "어떤 결제 수단? PG사는? 정기결제? 환불 정책은?"
→ 모호한 점 식별 후 사용자에게 확인 질문

4.5 Momus (비평가)

역할: 작업 계획을 검토하고 빈틈을 찾습니다.

Sisyphus가 계획을 세운 후:

→ Momus에게 "이 계획 검토해줘"
→ "3번 단계에서 에러 핸들링이 빠져있습니다"
→ 계획 보완 후 실행

4.6 병렬 작업 = 핵심 강점

사용자: "NextAuth로 소셜 로그인 구현해줘" 동시 발사 (병렬): ├─ Explore #1: "기존 인증 코드 패턴 검색" ├─ Explore #2: "기존 API 라우트 구조 검색" ├─ Librarian #1: "NextAuth v5 공식 문서 조사" └─ Librarian #2: "소셜 로그인 구현 예시 검색" → 4개 결과 동시 수집 → 최적의 구현 시작

05 카테고리 시스템

작업을 위임할 때 카테고리에 따라 최적화된 AI 모델이 배정됩니다.

카테고리용도예시
visual-engineeringUI, 디자인, CSS, 애니메이션"대시보드 레이아웃 만들어줘"
ultrabrain복잡한 로직, 알고리즘"최단 경로 알고리즘 구현해줘"
deep자율적 리서치 + 구현"성능 병목 찾아서 최적화해줘"
artistry창의적 문제 해결"독특한 인터랙션 디자인"
quick간단한 수정"오타 수정", "import 추가"
unspecified-low기타 간단 작업분류 어려운 간단 작업
unspecified-high기타 복잡 작업분류 어려운 복잡 작업
writing문서, 글쓰기"API 문서 작성해줘"
사용자가 알아야 할 것

직접 카테고리를 지정할 필요 없습니다! Sisyphus가 자동으로 판단합니다.

단, "예쁘게", "디자인", "레이아웃" 같은 키워드를 쓰면 → UI 특화 모델로, "복잡한 알고리즘", "최적화" 같은 키워드를 쓰면 → 논리 추론 특화 모델로 더 정확하게 라우팅됩니다.

07 작업 모드별 사용법

7.1 일반 모드 (기본)

그냥 말하면 됩니다. 가장 많이 쓰는 모드.

사용자: "Header 컴포넌트에 다크모드 토글 버튼 추가해줘"

1. 코드베이스에서 Header 컴포넌트 찾기 → 2. 기존 스타일/패턴 분석 → 3. 다크모드 토글 구현 → 4. 타입 검사 & 검증

적합한 상황: 명확한 단일 작업, 간단한 수정

7.2 울트라워크 모드 (자율 모드)

ulw + 작업 설명을 입력하면, 처음부터 끝까지 알아서 합니다.

사용자: "ulw 블로그 시스템 만들어줘"

1. 코드베이스 전체 탐색 → 2. 기술 스택/패턴 분석 → 3. 계획 수립 → 4. 구현 (여러 에이전트 병렬 활용) → 5. 테스트 & 검증 → 6. 완료될 때까지 반복

적합한 상황: 큰 기능 구현, "알아서 해줘" 스타일

7.3 Prometheus 모드 (정밀 모드)

Tab을 눌러 Prometheus를 선택하면, 먼저 인터뷰를 합니다.

[Tab → Prometheus 선택]

사용자: "e-커머스 결제 시스템 만들어줘"

Prometheus 인터뷰: "어떤 PG사? 정기결제? 환불 정책?" → 상세 계획 수립
/start-work → Atlas가 계획을 받아서 체계적으로 실행

적합한 상황: 복잡한 기능, 명확한 요구사항 정리가 필요할 때

7.4 모드 선택 가이드

작업 복잡도? │ ├─ 간단 (1-2 파일) ──→ 일반 모드: "버튼 추가해줘" │ ├─ 보통 (3-5 파일) ──→ 일반 모드: "로그인 기능 만들어줘" │ ├─ 복잡 (많은 파일) ─┬→ 빨리 해줘 → ulw: "ulw 대시보드 만들어줘" │ └→ 정확하게 → Prometheus: Tab → 인터뷰 → /start-work │ └─ 초대형 프로젝트 ──→ Prometheus 필수

08 실전 활용 패턴

패턴 1: 새 프로젝트 시작

말하기: "Next.js 15로 블로그 프로젝트 만들어줘. ShadCN UI 쓰고, 다크모드 지원하고, 마크다운 포스트 지원해줘."

→ nextjs15-init + frontend-design 스킬 자동 로드 → 풀 프로젝트 생성

패턴 2: 버그 수정

말하기: "yarn dev 하면 이런 에러가 나: TypeError: Cannot read properties of undefined (reading 'map') at UserList (src/components/UserList.tsx:15:23)"

→ 해당 파일 분석 → 원인 파악 → 최소한의 수정 → 타입 검사로 확인

패턴 3: 리팩토링

말하기: "src/utils/helpers.ts가 500줄이 넘어서 관리가 안 돼. 기능별로 분리해줘."

→ Explore로 의존성 분석 → 모듈 분리 → import 경로 자동 수정

패턴 4: 외부 라이브러리 도입

말하기: "이 프로젝트에 Zustand 상태관리 도입해줘. 기존 Context API 코드를 마이그레이션해줘."

→ Librarian: 최신 문서 조사 + Explore: 기존 코드 분석 → 마이그레이션

패턴 5: 디자인 & UI 작업

말하기: "대시보드 페이지를 예쁘게 만들어줘. 차트, 통계 카드, 최근 활동 리스트 포함."

→ visual-engineering + frontend-design 스킬 → 고품질 UI → 반응형/다크모드 자동

패턴 6: 여러 작업 동시 요청

말하기: "1. README.md 한국어로 번역해줘 2. ESLint 설정 추가해줘 3. 테스트 설정 (Vitest) 해줘"

→ 3개 작업을 병렬로 동시 처리 → 각각 최적 모델로 라우팅

패턴 7: 조사 + 구현

말하기: "경쟁사 A, B, C 랜딩페이지 분석해서 우리 랜딩페이지 만들어줘."

→ Librarian: 경쟁사 분석 → landing-page-guide-v2 스킬 → 전환 최적화 랜딩페이지

10 플러그인 활용

Notion 연동

Notion과 연결되어 있어 직접 조작이 가능합니다.

"노션에서 '프로젝트 관리' 데이터베이스 검색해줘"
"노션에 새 태스크 만들어줘: 로그인 페이지 구현"
"노션에 이번 스프린트 회의록 페이지 만들어줘"

사용 가능한 Notion 기능:

  • 데이터베이스 검색 및 쿼리
  • 새 행(Row) 추가
  • 페이지 생성
  • 태스크 관리
  • 워크스페이스 검색

11 프로 팁 & 트릭

팁 1: "왜"를 말하면 품질이 올라간다
❌
"버튼 추가해줘"
✅
"사용자가 모바일에서 한 손으로 빠르게 저장할 수 있게, 화면 하단에 플로팅 저장 버튼 추가해줘"
팁 2: 기존 코드 언급하면 일관성이 올라간다
❌
"로그인 페이지 만들어줘"
✅
"기존 회원가입 페이지(src/pages/signup.tsx) 스타일이랑 똑같이 로그인 페이지 만들어줘"
팁 3: 독립적인 작업은 한번에 요청 → 병렬 처리

"1. Header에 알림 아이콘 추가 / 2. Footer 저작권 연도 변경 / 3. 404 페이지 만들기"
→ 3개가 독립적이므로 동시 처리 → 빠름!

팁 4: 에러 메시지를 그대로 붙여넣기
❌
"빌드 에러 나"
✅
"빌드하면 이런 에러 나: Module not found: Can't resolve '@/components/ui/button' at src/pages/index.tsx:3:1"
팁 5: 참고 자료 URL 제공

"이 디자인처럼 만들어줘: https://example.com/design
색상은 #3B82F6(파랑) 기반으로"

팁 6: 단계적으로 진행하기

1단계: "먼저 DB 스키마만 설계해줘" → 검토
2단계: "좋아, 이제 API 엔드포인트 만들어줘" → 검토
3단계: "이제 프론트엔드 연결해줘" → 검토

팁 7: 틀렸으면 바로 말하기

"아니, 그게 아니라 카드형 레이아웃으로 해줘"
"이 부분 롤백하고 다른 방법으로 해줘"

팁 8: 울트라워크(ulw)는 큰 작업에만
❌ 낭비
ulw 오타 수정해줘
✅ 적합
ulw 관리자 대시보드 만들어줘

12 자주 하는 실수 & 해결법

실수 1: 너무 모호하게 말하기
❌
"개선해줘" → 뭘 개선? 성능? UI? 코드 품질?
✅
"이 컴포넌트의 렌더링 성능을 개선해줘. 리스트가 1000개 넘으면 느려져."
실수 2: 한번에 너무 많은 것 요청하기
❌
"전체 앱 TypeScript 마이그레이션 + 테스트 + CI/CD + Docker + 문서"
✅
"먼저 src/utils/ 폴더만 TypeScript로 변환해줘"
실수 3: 컨텍스트 없이 에러 물어보기
❌
"왜 안 돼?"
✅
"yarn build 하면 이 에러가 나: [에러 메시지 전체 복붙] Node 20.x, Next.js 15 환경이야."
실수 4: 기존 패턴 무시하기
❌
"Redux로 상태관리 추가해줘" (프로젝트가 이미 Zustand 사용 중)
✅
"기존 Zustand 패턴으로 장바구니 상태 추가해줘"
실수 5: 중간에 끊기
❌
작업 중간에 새 요청 → 이전 작업 미완성
✅
현재 작업 완료 확인 → 다음 작업 요청
또는: "이건 잠깐 멈추고, 먼저 이거 해줘"
